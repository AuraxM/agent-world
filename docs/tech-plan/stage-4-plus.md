# Stage 4+ · 候选功能（P1）

## 状态

Stage 4+ 是 **MVP 上线后**的扩展规划，不在初版交付范围内。

**启动条件**：在 Stage 3 上线后做一轮真实玩家测试（≥ 10 名），根据反馈再决定优先级。重点观察：
- 玩家是否觉得"事件投放"是足够的交互？
- 是否有人反复请求"对话/扮演"功能？
- 创作型玩家是否需要更强的模板/分享能力？

---

## 候选功能列表

### A. 只读快照分享（高优先级候选）

**用户价值**：玩家把自己导演出的精彩故事分享给他人观看。

**功能**：
- 玩家选择某段历史（tick 范围）→ 生成只读分享链接
- 他人无需登录即可访问 URL → 看到完整地图、NPC、事件流
- 不可修改、不可继续模拟

**技术要点**：
- 导出快照 + 事件日志 → 上传到对象存储（如 R2/S3）
- 短链接生成 + 有效期管理
- 只读视图路由 `/share/[shareId]`，复用主视图组件
- 防滥用：每用户限流（≤ N 个分享/天）

**预估工作量**：1 周

---

### B. 多世界比较

**用户价值**：玩家创建分支后想并排对比"如果当时 X 没发生..."

**功能**：
- 同时打开 2–3 个世界并排观察
- 时间游标可"联动"或"独立"
- 标注差异点（哪个事件在 A 世界发生，B 世界没有）

**技术要点**：
- 主视图改造为支持多列布局
- 单个 tick 内的差异检测（diff snapshot）
- 性能：3 个世界同时活跃需评估

**预估工作量**：1.5 周

---

### C. 永久死亡

**用户价值**：让世界有重量感，玩家投放危险事件时承担真实后果。

**功能**：
- NPC 可真死（重伤未治、年老、玩家投放死亡事件）
- 死亡角色不再决策，但保留在历史与他人记忆中
- 关联 NPC 触发"丧亲"长期状态 + 性格漂移

**技术要点**：
- Character 增加 `status: 'alive' | 'gone' | 'dead'` 字段（gone 是 MVP 的"退场"，dead 是新增）
- 死亡事件的 scope 自动扩展到所有"关系深度 ≥ X"的 NPC
- UI 上死亡 NPC 用灰色 + 黑纱标记
- 关系矩阵中死亡 NPC 不可新增关系

**预估工作量**：1 周

**注意**：永久死亡引入后需要考虑用户心理——是否提供"撤销死亡"？建议提供"分支挽救"按钮，引导用户在分支里救人，而不是直接 undo。

---

### D. 内容扩展（持续工作）

不是一个独立 Stage，而是持续投入：

**事件模板扩充**：
- 节日（春节、生日、毕业季）
- 灾难（火灾、地震、瘟疫）
- 神迹 / 奇幻（魔法、超自然）
- 经济类（涨价、缺货、丢钱）

**剧本预设**：
- "侦探案件"（嫌疑人 + 受害者 + 线索 NPC）
- "三角恋"（性格故意冲突的三人组）
- "选举"（小镇竞选剧情）
- "末日生存"（资源稀缺世界）

**NPC 模板库**：
- 用户上传 / 共享单个 NPC 卡
- 评分 / 评论系统

**预估工作量**：滚动发布，每月新增 3–5 个

---

### E. 国际化 i18n

**用户价值**：扩大用户群（英文世界）。

**功能**：
- 至少中英双语 UI
- UI 文案与 NPC 内容文案分离
- NPC 使用语言可独立于 UI 语言（中文 UI 看英文 NPC）
- 数字/日期格式跟随地区

**技术要点**：
- 引入 next-intl 或类似库
- 文案抽取到 `locales/{zh,en}.json`
- LLM prompt 也要双语化（system prompt 与 NPC 输出语言对齐）
- shadcn/ui 组件文案替换

**预估工作量**：1.5 周

---

### F. 移动端深度优化

**用户价值**：碎片时间挂机型玩家的核心场景。

**功能**：
- Tab 化布局（世界 / 事件 / 角色 / 我）
- 简化操作（编辑器、关系图谱仅桌面）
- 推送通知（重要事件、快进完成）
- 离线快照查看

**技术要点**：
- responsive 全面 review
- PWA 化（Service Worker + manifest）
- Web Push 接入（通知）

**预估工作量**：2 周

---

### G. NPC 自主创造（高级涌现）

**用户价值**：NPC 不只是反应，还能主动改变世界。

**功能**：
- NPC 可写信、生孩子、建房子、创业
- 这些行动会扩大世界的状态空间
- 例：一个高"创造性"的 NPC 在工坊里"发明了风车" → 世界出现新节点 / 新物品

**技术要点**：
- 行动类型扩展（`create_item`, `establish_node`, `procreate`）
- 创造结果由 LLM 生成 + 引擎校验
- 产物入库（新增 nodes / characters / items）
- 防滥用：限制 per-tick 创造次数

**预估工作量**：2 周（高复杂度）

---

### H. 集体行动 / 群体涌现

**用户价值**：让大场面（聚会、暴动、集会）真正像群像。

**功能**：
- 群体逃难、合唱、起哄、游行
- 不是每人各自决策，而是有"群体涌现"机制
- 群体内的 NPC 部分让渡决策权给"群体"

**技术要点**：
- 引入 `Group` 概念（临时聚集体）
- LLM 一次为整个群体决策大方向，再派生个体细节
- 性能优化：群体决策替代 N 次个体决策

**预估工作量**：2 周

---

### I. NPC 撒谎 / 隐瞒

**用户价值**：增强真实感（人会撒谎）。

**功能**：
- NPC 可在对话中输出与真实状态不符的内容
- 撒谎成功率受 honesty 维度影响
- 被识破后关系恶化

**技术要点**：
- LLM 输出增加 `is_lying` 标记 + 真实意图
- 关系系统中加入"可信度"折扣
- 可观测性：玩家在内心独白中能看到"撒谎"标签

**预估工作量**：1 周

**风险**：会大幅增加事实一致性的复杂度，需要充分测试。

---

### J. 历史摘要与"长记忆"压缩

**用户价值**：让长期模拟（30+ 天）的世界依然连贯。

**功能**：
- 每游戏周自动由 Haiku 生成"本周纪事"
- 旧的短期记忆归并为"那段时间发生过 X"概括
- 历史摘要可作为决策的远景上下文

**技术要点**：
- 周末（每 168 tick）调用一次 Haiku 摘要
- 摘要结果存储 + 注入未来 prompt
- 玩家可阅读"本周纪事"作为叙事辅助

**预估工作量**：1 周

---

## 优先级建议

基于"用户价值 / 工作量"评估，建议顺序：

| 优先级 | 功能 | 理由 |
|---|---|---|
| P1.1 | A. 只读快照分享 | 病毒传播潜力大，工作量小 |
| P1.2 | C. 永久死亡 | 用户呼声大，体验深度提升明显 |
| P1.3 | D. 内容扩展（事件模板）| 滚动迭代，每次小投入 |
| P1.4 | J. 历史摘要压缩 | 解决长期模拟瓶颈，是其他 P1 的基础 |
| P2.1 | E. 国际化 | 拓展市场，但中文用户先打磨 |
| P2.2 | F. 移动端 | 取决于用户使用数据 |
| P2.3 | B. 多世界比较 | 高级玩家需求 |
| P3 | G/H/I | 复杂度高，需先验证基础玩法稳固 |

---

## 不在规划内的功能（明确不做）

- **玩家附身扮演**：已确认产品定位是"导演"，不做扮演
- **玩家与 NPC 直接对话**：同上
- **多人协作模式**：MVP 阶段不考虑（成本、同步、伦理复杂度高）
- **NPC 觉醒元叙事**（NPC 知道自己是 NPC）：会破坏沉浸感
- **强成就 / 任务系统**：与"开放沙盒"定位冲突

---

## 评估机制

每个 Stage 4+ 功能上线后：
- 收集 1–2 周用户反馈
- 关注核心指标：留存、平均会话时长、世界创建数、事件投放数
- 与"基线（Stage 3 上线后）"对比，决定是否继续投入相关方向

---

## Stage 4+ 完成后的形态

agent-world 应成为：
- 一个"小说生成器" + "上帝模拟器"的混合产品
- 创作者社群（共享模板、剧本、精彩世界片段）
- 长期叙事可承载（数十游戏日不崩坏）
- 多平台（桌面 + 移动）
- 多语言

但所有这些都不应破坏 MVP 的核心体验：**玩家以导演身份，向世界投放事件，观察 NPC 涌现的差异化反应**。
